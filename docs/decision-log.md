# Registry Design Decision Log

Status: historical record of the brainstorm that shaped `specs/001-registry-foundation/spec.md`. This document is not itself governing — if it and the spec disagree, the spec wins. Kept for provenance: why the shape is what it is, not just what the shape is.

Date: 2026-07-03

## Context

Traverse (the runtime) already had an in-repo registry engine (`crates/traverse-registry` in `traverse-framework/traverse`) covering public/private overlay resolution, semver rules, dependency locking, connector registration, and composability metadata (governed by that repo's specs 005, 007, 011, 034–037, 039, 041, 043). That engine already assumed a "public registry" existed as a data source, but nothing actually served that role. This log captures the decisions made to stand up that role as a new, dedicated repo: `traverse-framework/registry`.

## Decisions

1. **Split boundary**: `traverse-contracts` stays in `traverse-framework/traverse`; `traverse-registry` (the crate, plus all capability content) moves to this repo. Rationale: `traverse-contracts` is the schema every consumer (runtime, CLI, MCP, this registry) depends on symmetrically — one canonical home avoids two repos both claiming to define "the schema." See `traverse/specs/051-registry-extraction/spec.md` for the formal migration record.

2. **CLA**: required org-wide, starting now (before this repo or `traverse` accumulates outside contributors), to preserve the option to relicense or commercially license the codebase later without needing retroactive consent from every contributor. Governed centrally in `traverse-framework/.github`.

3. **Architecture — git-based static index, not a hosted service, for v1**: capability records are files in this repo, validated and quality-gated by CI, merged via reviewed PRs — the same shape as crates.io's index repo or Homebrew-core. Chosen because: (a) it's free to run (GitHub Actions is free for public repos — no server, no database, no bill regardless of usage), and (b) it fits the existing spec-gated PR governance model this org already uses. Explicitly designed so a hosted API layer can be added later without a breaking schema migration — the resolve/fetch interface and the record schema are meant to be storage-agnostic from day one.

4. **Future hosted layer, when built**: serverless/edge, pay-per-use. Read path (resolving/fetching capabilities — the overwhelming majority of traffic in any registry) sits behind a CDN/edge cache; write path (publishing) runs on pay-per-request compute; artifact storage uses cheap object storage. The goal: running cost scales with actual usage, never a fixed bill for idle capacity.

5. **Schema reserves `owner` and `namespace` fields now**, defaulted to a "core" identity and an unscoped namespace, even though the only accepted publisher in v1 is the core team via PR review. Rationale: adding real third-party/dev publishing later without these fields would require a breaking migration of every existing record; reserving them now costs almost nothing.

6. **Sync/consume model**: `traverse-cli registry sync` (CLI, lives in `traverse-framework/traverse`) fetches the latest published index into local durable workspace state. The runtime (`traverse-registry` crate, wherever it ends up being consumed from) only ever resolves against that local synced state — zero live network dependency at execution time. This reuses the exact pattern `traverse`'s spec 046 already established for app registration (CLI writes durable local state, runtime reads it, never calls a live service).

7. **Publish mechanism**: merging a PR to `main` in this repo triggers CI to build a versioned aggregated index artifact, published as a GitHub Release. `sync` fetches that artifact, not the raw git tree. This reuses the shape of `traverse`'s existing spec 048 (tag → build → publish pipeline for its own crates), rather than inventing a new mechanism.

8. **Quality gate — deterministic + AI-advisory, not AI-blocking**: schema validation, semver-bump-vs-actual-diff checks, digest integrity, namespace-collision checks, and dependency-resolvability checks are deterministic and block merge. A separate AI pass flags likely semantic duplicates (capabilities that overlap in effect but not in name/tags) and capability-boundary quality issues (e.g. a CRUD wrapper disguised as a capability) as PR comments — advisory only. This is required to be advisory, not blocking, because the constitution (Principle II, inherited from `traverse-framework/.github`) already requires explicit manual approval before any contract publication; the AI pass informs that human review, it doesn't replace it.

9. **CLI automates the PR flow**: `traverse-cli capability publish` (implemented in `traverse-framework/traverse`, not here) validates a contract locally, computes the artifact digest, and uses git + `gh` to open a PR against this repo automatically — the equivalent of Homebrew's `brew bump-formula-pr`. The human review/approval gate on the PR is unchanged; only the mechanical git steps are automated.

10. **Repo layout**: `capabilities/<namespace>/<id>/<version>/contract.json` (+ artifact references alongside). Path-encodes identity so two capabilities structurally cannot occupy the same path (namespace collisions become impossible, not just checked), and a new version is always a new file — the filesystem itself enforces the immutability rule already required by `traverse`'s spec 005, instead of relying purely on CI logic to catch violations.

11. **Artifact storage**: WASM binaries are **not** committed into this git repo directly (git handles binary blobs and unbounded history growth badly, which actively fights an append-only immutable-version model). They live as GitHub Release assets; the git-tracked contract stores only a digest + release URL. Free to host, keeps the repo itself small and fast indefinitely.

12. **Deprecation**: a `cargo yank`-style mechanism — a version is never edited or deleted, but a separate deprecation-flag file marks it excluded from range resolution (`^1.2.0`-style) while still resolvable by an exact pin. This is the standard, well-proven shape every major package registry has converged on for exactly this problem, and it doesn't touch immutability.

13. **Tracking**: this repo gets its own GitHub Project board (Project 3, "Registry" — already existed, needed cleanup of 7 stray unrelated `traverse` issues that had been swept in incorrectly).

14. **Semver rules**: inherited as-is from `traverse`'s specs 005/037/043, moving with the crate rather than being redesigned.

15. **Repo naming across the org** (`traverse`, `reference-apps`, `website`, `registry` — lowercase-kebab-case, matching the dominant OSS convention and the org's existing crate-naming style) — decided in the same session, but tracked separately in `traverse/docs/repo-migration-plan.md`, not part of this repo's scope.

16. **Governance consolidation**: constitution, NFRs, quality standards, antipatterns, compatibility policy, exception process, CLA, and the spec-alignment CI script were pulled out of `traverse-framework/traverse` into a new shared repo, `traverse-framework/.github`, rather than copied into this repo. Every repo (including this one) keeps a thin pointer to a pinned governance version instead of a duplicated copy — see `.specify/memory/constitution.md` in this repo.

17. **AI-advisory review tooling refinement, then reverted (see 18)**: briefly switched the pass to a `.claude/skills/capability-review` Claude Code skill invoked via `claude -p` in CI, authenticated with a subscription token (`CLAUDE_CODE_OAUTH_TOKEN`, from `claude setup-token`) instead of a metered `ANTHROPIC_API_KEY`. The intent was avoiding per-call API billing when the org already carries a Claude subscription.

18. **Reverted decision 17 — subscription auth shares a real quota with personal usage**: `CLAUDE_CODE_OAUTH_TOKEN` authenticates as the account's actual Claude subscription, and Pro/Max/Team plans share **one unified usage limit across all Claude Code access** — interactive terminal sessions, IDE usage, and any CI job using that same token all draw from the same pool. For the repo owner, who actively depends on that plan for real interactive work, this meant CI runs could plausibly push personal usage into hitting plan limits sooner, or during, active interactive sessions. Reverted back to the original metered `ANTHROPIC_API_KEY` (Claude Console workspace key) design: costs isolated, billed separately, zero risk of touching personal usage. The `capability-review` skill file was removed since it's not used by the reverted implementation; the prompt logic lives inline in `scripts/ci/ai_advisory_review.py` instead. Spec 004's FR-001–004 are unaffected by this reversal, same as decision 17 — no normative requirement changed either way.

19. **`ANTHROPIC_API_KEY` deliberately not provisioned (as of 2026-07)**: after reverting decision 17, the repo owner explicitly said they don't want to allocate budget for a metered `ANTHROPIC_API_KEY` right now. The `ai-advisory-review` CI job is left wired up and correct (see `scripts/ci/ai_advisory_review.py`) but runs dormant: with no secret configured, `call_claude` raises immediately and every PR touching `capabilities/**/contract.json` gets the `DEGRADED_COMMENT` advisory notice instead of a real review. This is expected, not a bug -- do not "fix" it by requesting a key, by reintroducing subscription-token auth (see decision 18 for why that was reverted), or by making the job block merges to compensate. It stays this way until the repo owner decides to fund it.

20. **Bundle `scope: public/private` reconciled with this registry (2026-07-06)**: the pre-existing `traverse-registry` crate concept (traverse spec 005 FR-006/007: a public tier plus a private/local overlay with `PreferPrivate` lookup precedence, keyed as `(scope, id, version)`) and this repo's `namespace`/`owner` fields are **orthogonal axes that both stay** — resolution tier vs publisher identity. What was pinned down: **the public tier of any runtime's local state is populated only by `traverse-cli registry sync` from this repo's published index; "public" means "published in `traverse-framework/registry`," nothing else.** Consequences (normative traverse-side, tracked in `traverse-framework/Traverse#548`): `scope: private` keeps overlay behavior and additionally becomes a publish-refusal latch (`capability publish` refuses private-bundle content, npm `"private": true`-style); local `bundle register` with `scope: public` is rejected once sync exists (dev/testing is fully served by the private overlay, which already wins lookup — the cargo answer, where "inserting into your local copy of the crates.io index" isn't an operation that exists); a private-overlay record shadowing a synced public identity is allowed with a warning (cargo `[patch]`-style). **Why**: a survey of npm, cargo/crates.io, Go modules, and Docker/OCI showed no mature ecosystem lets an artifact author declare "I am public" as routing metadata — publicness is always a fact about the source a record came from, and the npm/PyPI dependency-confusion attack class is exactly what leaving two writers of one public namespace enables. The alternative options — leaving the two meanings of "public" unreconciled, or replacing declared `scope` with derived provenance — were rejected respectively as "guaranteed `ImmutableVersionConflict` at sync time" and "a breaking manifest-schema migration that buys nothing Option A doesn't." Team-shared private registries (multi-source sync with per-namespace routing, npm-`.npmrc`-style) were explicitly deferred but designed-for: the single-source rule generalizes to a source list additively. Registry-side companion: `specs/006-public-scope-and-identity` (Draft).

21. **Consumer-side resolution goes dual-mode; artifacts fetched at registration time (2026-07-06)**: once `registry sync` (traverse #542) exists, component manifests support two mutually exclusive capability reference forms — the existing `contract_path` (+ `wasm_binary_path`/`wasm_digest`), which keeps meaning "this component bundles its own capability" and registers it into the **private overlay** (consistent with reference-apps' existing `workspace_defaults.registry_scope: "private"`), and a new `registry_ref: {namespace, id, version_range}` resolved from the sync-populated public tier. **Why dual-mode**: it is the convergent prior-art shape (cargo path-vs-registry dependencies, Go `replace`, npm `file:`), it is purely additive (no existing manifest breaks), and the rejected alternatives both broke the app-local development model — registry-only references would force publishing before an app could even register its own capability, and a separate override file adds indirection to what is today the common case. **Artifact fetch timing**: `sync` fetches `index.json` only; WASM artifacts download at app-registration time into a local content-addressed, digest-verified cache. This keeps sync cheap (bandwidth scales with what you use, not with the registry), while preserving the constitutional zero-live-network-at-execution rule — registration is a CLI-time operation, execution never touches the network. Normative home: the traverse-side spec under `traverse-framework/Traverse#548`; reference-apps adoption tracked as `reference-apps#97`, gated on that work plus seed content.

22. **Seed content: `traverse-starter.process` 1.0.0, published as-is via manual PR; artifacts hosted in this repo (2026-07-06)**: the registry's first real content is the one genuine capability contract the org has (`trace-explorer` turned out to have no contracts at all — it is a React client only), published unchanged: namespace `traverse-starter`, existing `owner` object, existing digest. **Why as-is**: re-namespacing to `core` would break identity continuity (when reference-apps later flips to `registry_ref`, the identity it references should be the one it already uses) and would skip exercising the non-core namespace path; a synthetic hello-registry capability would prove the pipeline on throwaway content when real content exists. **Why a manual PR, before the #543 CLI**: the CLI only automates git mechanics, so publishing without it proves the pipeline is CLI-independent and gives #542/#543 real content to build and test against — seeding first inverts the dependency the right way. **Why artifacts live in this repo's releases** (tag scheme `artifacts/<id>-<version>`): this repo owns the immutability guarantee, so a published record's integrity must not depend on another repository's release hygiene; the wasm is uploaded as a release asset here and the published contract copy gains the `artifact: {digest, url}` field (the source contract in traverse stays untouched — the registry copy is the publication record). Governed by `specs/007-artifact-hosting` (Draft), which also closes the "which release, uploaded when, by whom" gap spec 001 FR-007 left open.

23. **Draft specs are tracked in a `draft_specs` array, not in `specs[]` (2026-07-06)**: the session's ground rule was "new specs enter `approved-specs.json` as Draft; approval is the repo owner's explicit, standalone act." The vendored spec-alignment gate, however, fails CI if **any** `specs[]` entry has `status != approved` (it validates every record, not just declared ones). Rather than forking the vendored governance script (contradicting decision 16) or leaving drafts untracked in the governance file, drafts are registered in a separate top-level `draft_specs` array in the same file — the gate reads only `schema_version` and `specs`, so drafts are visible and machine-readable without breaking CI. Approval = the repo owner (never an agent) moving the record into `specs[]` with `status: approved` / `immutable: true` and the `will_govern` list becoming `governs`. If governance later standardizes a draft-tracking convention upstream in `traverse-framework/.github`, this repo adopts it and drops the local convention.

24. **Co-brainstormed specs are approved by the brainstorm itself (2026-07-06)**: the repo owner stated that specs produced from a joint brainstorm session with the owner do not require a separate, standalone sign-off act — the owner's participation in and conclusion of the brainstorm *is* the approval. Exercised immediately for specs 006 and 007 (both designed question-by-question with the owner in the 2026-07-06 planning session), which move to `status: approved` in the same PR that introduced them as drafts. The no-self-approval rule (AGENTS.md) is unchanged in substance: an agent still may never approve a spec on its own judgment — what changed is that for co-brainstormed specs the sign-off is given in-session rather than as a later, separate act. The `draft_specs` convention (decision 23) remains in place for any spec an agent drafts without the owner having settled its design in-session.

Costs are covered by the owner's existing Claude subscription *interactively*, which does not have the CI-shares-personal-quota problem that killed decision 17 (the review runs inside a session the owner is already using, not as unattended CI burn). The CI job stays exactly as decision 19 left it: wired, correct, and dormant, posting the `DEGRADED_COMMENT` notice — spec 004's FR-001–004 govern the CI behavior and are unchanged; the skill is additive session tooling fulfilling the same advisory function through a different channel. Human judgment remains the only gate either way. Skill creation tracked as its own registry ticket.

27. **`index.json` gains contract provenance fields (`contract_digest`, `contract_url`), agent-drafted as `specs/009-contract-metadata-in-index` (Draft), pending owner sign-off (2026-07-14)**: registry ticket #44, tracing to the already-approved `traverse-framework/traverse` spec `063-registry-contract-materialization` FR-001/FR-002, asked for every public index record to carry a resolvable, digest-verified contract reference alongside the existing artifact reference — closing the gap where Traverse's `registry_ref` consumer path (Traverse #551) could otherwise register a capability from a verified WASM artifact without ever fetching or checking its behavioral contract. Implemented in `scripts/ci/build_index.py`: `contract_digest` is `sha256:<hex>` of the exact `contract.json` bytes read at build time (the same immutable file the index is built from, so no separate verification fetch is needed at build time — the "fetch" and the "verify" share one read); `contract_url` is a `raw.githubusercontent.com` link pinned to the current build's `source_commit`, valid because `capabilities/<namespace>/<id>/<version>/contract.json` paths are structurally immutable (decision 10) — any commit on `main` at or after a file's introduction serves byte-identical content, so pinning to the *current* build's commit (rather than resolving each file's original introducing commit via `git log`) is sufficient and cheaper. Also fixed a latent bug while touching this code: `build_index.py` previously silently `continue`d past an unreadable/corrupt `contract.json`, silently shrinking the public index — it now hard-fails the build with a stable `index.contract_unreadable` error instead, since a silently-dropped record is the same failure class (an unverifiable record reaching a consumer) this ticket exists to close. Per this repo's no-self-approval-of-specs rule, `specs/009-contract-metadata-in-index` stays `Draft` in `draft_specs` — the code changes are implemented and tested against it (precedent: decision 22 implemented spec 007 while it was still Draft), but the spec's `Approved`/immutable status requires the repo owner's own explicit, standalone sign-off.

29. **Crate publish pipeline split out of #9 as its own prerequisite (`specs/010-crate-publish-pipeline`, approved), a placeholder `traverse-registry` 0.0.1 reserves the crates.io name immediately, fresh-start history for the eventual move, `traverse`-side cutover is one atomic PR pinning an exact version (2026-07-18, `/brainstorm` with the repo owner, cross-repo with `traverse#627`)**: resuming #9 (decision 26) surfaced that this repo has zero Cargo/Rust structure today, and that the two tracking tickets' DoD both assumed "depend on the externally-published crate" without ever ticketing the work of making that crate publishable. Six questions, decided one at a time: (a) **history** — the eventual crate-content move imports no git history into this repo; a fresh-start snapshot is taken instead, since `traverse`'s own git log already preserves the pre-extraction history permanently (path-filtered `git log -- crates/traverse-registry`), and grafting an unrelated-histories merge into this repo's own clean, curated commit graph would cost real clarity for no discoverability this repo actually needs. (b) **ticketing** — the scaffold + pipeline work is a new prerequisite ticket in this repo (not folded into #9), so a broken publish pipeline and a broken content move are never the same failure to debug; #9 stays blocked on this ticket resolving first. (c) **spec coverage** — a new spec (`010-crate-publish-pipeline`) governs the Cargo/CI surface specifically (`Cargo.toml`, `crates/traverse-registry/`, `.github/workflows/publish-crate.yml`, `scripts/ci/publish_crate.sh`), kept separate from the specs being re-adopted for #9 (005/007/011/034–037/039/041/043), which govern registry/capability semantics inherited from `traverse` — a different concern that was never meant to cover build tooling. Per decision 24 (co-brainstormed specs are approved by the brainstorm itself), spec 010 goes straight to `approved` in the same PR that introduces it. (d) **name reservation** — the prerequisite ticket's placeholder crate publishes under the exact final name, `traverse-registry`, at `0.0.1`, immediately once the pipeline works — not a throwaway test name reserved later — because this org already lost the identical race once (`traverse-cli` → `traverse-cli-rs`, `traverse` PR #743, days before this session) and there is no reason to reopen that exact risk window. 30. **Registry-side content governance for the extracted crate resolved as a hybrid: 2 specs re-adopted verbatim, 19 covered by one blanket inherited-governance spec (2026-07-18, `/brainstorm` continuation of entry 29)**: entry 29 covered the publish-pipeline prerequisite; a follow-up question surfaced while double-checking ticket completeness -- of the (now confirmed) 21 `traverse` specs whose `governs` includes `crates/traverse-registry/`, only `005-capability-registry` and `011-event-registry` are actually registry-exclusive there (their only other listed path, `crates/cogolo-registry/`, is dead pre-rename cruft that doesn't exist on disk); the other 19 also govern `traverse-runtime`, `traverse-cli`, or `traverse-contracts`, none of which are moving. Three options were weighed: full re-adoption of all 21 (maximum traceability, but ~19 specs' worth of real synthesis work extracting registry-relevant FRs from requirements that often describe a registry/runtime or registry/CLI interaction as one inseparable requirement, plus permanent dual-copy drift risk), a single fully blanket spec covering all 21 including the two clean ones (simplest, but throws away free precision on 005/011 for no reason), or a hybrid. **Decided: hybrid.** `011-capability-registry-adoption` and `012-event-registry-adoption` are verbatim copies of `traverse`'s 005 and 011 (full FR-level content preserved, zero synthesis needed since they were already registry-exclusive) -- `013-inherited-registry-governance` is a new, deliberately non-restating spec that names the other 19 source specs, states the extracted crate's behavior is governed *as-is* from its state at extraction, and requires (FR-002) that any future behavior change go through a new registry-side spec rather than an edit to the inherited list. **Enforcement timing changed mid-session**: all three were initially drafted to sit unregistered (approved in content per decision 24, but withheld from `specs/governance/approved-specs.json`) per entry 29's decision 26(f)-consistent caution against declaring governance over a path that didn't exist yet. Before this PR was opened, the repo owner directly authored and merged #52 (implementing #50: the Cargo scaffold, placeholder `traverse-registry` 0.0.1 package, and publish-CI pipeline from spec 010) -- `crates/traverse-registry/` now physically exists, so all three specs register immediately instead, per the same reasoning that already governs spec 010 registering against a path whose content doesn't yet fully satisfy it (precedent: decision log entry 22, spec 007 implemented while still Draft -- spec-then-implementation is the normal order, not a claim of already-satisfied FRs). `traverse#627` and `traverse` decision log entry 29's snapshot are unaffected by this entry -- the 19-spec source-list correction (10 originally named in #9's text, 21 actually current) belongs to entry 29's scope, already recorded there and in `traverse#627`'s DoD.
